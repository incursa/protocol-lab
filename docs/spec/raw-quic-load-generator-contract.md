# Raw QUIC Load Generator Contract

Phase 6 defines raw QUIC transport scenarios without pretending every scenario
has runnable validation coverage.

## Current State

- Scenario family: `quic.transport`
- Scenario files: `scenarios/quic/transport/*.yaml`
- Load tool: `load-tools/quic-go-raw-load.yaml` invokes the public reference
  raw QUIC test executor through the generic process load-tool contract. Hosted
  lab runs should prefer the package v2 `test-executor` package so the worker
  launches a package-local executable instead of relying on a source checkout.
- Enabled load-validated scenarios:
  - `quic.transport.multiplex.100x64kb`
  - `quic.transport.duplex-streams`
- Validator behavior: other raw QUIC transport scenarios still return
  explicit `unsupported` results with a reason.
- Metrics model: raw load JSON is parsed into the common result metrics,
  including bytes sent, bytes received, completed streams, failed requests,
  and timeout requests.

## Activation Gate

Additional raw QUIC scenarios can be enabled only after a scheduled slice adds
scenario-specific gates equivalent to the enabled duplex/multiplex slice:

- target process/container contract coverage for that scenario
- TLS/ALPN and certificate handling for the selected target/load pair
- scenario-specific validation for handshake, stream throughput, or connection
  churn as applicable
- raw stdout and stderr preservation
- qlog or equivalent protocol evidence when available; when unavailable the
  result must say so rather than fabricating protocol evidence
- parser behavior that keeps `parsedMetricsAvailable=false` when parsing fails

## Runner Boundary

The runner keeps raw QUIC support behind generic manifest, validation,
load-tool, artifact, and parser contracts. Raw QUIC traffic is generated by
the selected target and load tool, not by runner-linked Incursa QUIC
assemblies. The runner does not emit successful raw QUIC benchmark data unless
validation passed for the selected implementation and scenario.
