# Architecture — Artifact Model

**Status:** Implemented (deterministic layout, collision-proof cell identity, 60+ artifact path types, artifact generation complete)

## Purpose

The artifact model defines where and how ProtocolLab stores the evidence
produced during validation and benchmarking. Artifacts provide proof of what
happened — raw load-tool output, server logs, protocol traces, environment
metadata, and parsed results. The layout is deterministic, predictable, and
designed for auditability.

## Assembly

- **Project:** `src/Incursa.ProtocolLab.Model`
- **Namespace:** `Incursa.ProtocolLab.Model`
- **Key types:** `ArtifactLayout`, `ArtifactPaths`

## Artifact Layout

All artifacts live under `.artifacts/runs/{runId}`. Each run cell has a
deterministic subdirectory:

```
.artifacts/runs/{runId}/
  implementations/
    {implementationId}/
      {scenarioId}/
        {protocolId}/
          {executionProfileId}/
            {networkProfile}/
              {loadProfileId}/
                c{connections}-s{streams}-r{repetition}/
```

This layout makes artifact paths predictable before execution and stable
across runs. Tools that consume artifacts (report generators, analysis
scripts, dashboards) can locate any artifact by constructing the path from
run metadata.

The path segments come from `RunCellIdentity`. The identity includes the
implementation id, scenario id, normalized protocol id, execution profile id,
network profile, load profile id, connection count, stream count, and
repetition. Each segment is sanitized so file-system collisions do not occur
when ids contain punctuation.

### Run-Level Artifacts

| Path | Content |
|------|---------|
| `run-metadata.json` | Host OS, .NET runtime, Docker backend, git commit |
| `run-events.jsonl` | Structured event log for the entire run |
| `summary.md` | Markdown summary of results |
| `aggregate-results.json` | Machine-readable aggregate of all cell results |
| `index.md` | Compact run index (generated by `New-ProtocolLabRunIndex.ps1`) |

### Per-Cell Artifacts

Each run cell directory contains:

| Path | Content |
|------|---------|
| `result.json` | Full `BenchmarkResult` or validation result |
| `validation.json` | Validation status and observations |
| `validation-results.json` | Per-cell validation results (validation-only runs) |
| `protocol-proof.json` | Protocol negotiation proof (method, versions, response) |
| `protocol-proof.stdout.txt` | stdout from protocol proof command |
| `protocol-proof.stderr.txt` | stderr from protocol proof command |
| `load-tool.stdout.txt` | Raw stdout from load tool invocation |
| `load-tool.stderr.txt` | Raw stderr from load tool invocation |
| `load-tool-command.txt` | Executed command line (or Docker command) |
| `load-tool-output.json` | Parsed load-tool JSON output (when available) |
| `server-stdout.txt` | Target server stdout (when captured) |
| `server-stderr.txt` | Target server stderr (when captured) |
| `server-log.txt` | Target server log file (when exported) |

There is no separate `cell.json` artifact today. The cell identity is already
encoded in the directory structure and in the result payload. If a future
`cell.json` is added, it should mirror `RunCellIdentity` rather than become a
second identity source.

### Diagnostic Artifacts

| Path | Content |
|------|---------|
| `diagnostic-target.json` | Resolved diagnostic target identity |
| `counter-stdout.txt` | `dotnet-counters` stdout |
| `counter-stderr.txt` | `dotnet-counters` stderr |
| `counter-raw.csv` | Raw counter samples (when available) |

### Trace Artifacts (Modeled)

| Path | Content |
|------|---------|
| `qlog/` | qlog trace directory (when target exports qlog) |
| `sslkeylog.txt` | SSL key log (when target exports keys) |
| `pcap/` | Packet capture directory (future) |

qlog paths exist in the artifact layout for Docker h2load when the image
proves `--qlog-file-base`. SSL key log and pcap paths are modeled, but their
capture remains deferred until a target exports them.

## ArtifactPath Generation

`ArtifactLayout` exposes static methods for constructing deterministic paths:

```csharp
string baseDir = ArtifactLayout.GetCellDirectory(outputRoot, runId, cell);
```

Individual artifact paths are built by appending the artifact filename to
the cell directory.

## Principles

1. **Determinism.** Artifact paths depend only on run parameters (run ID,
   implementation ID, scenario ID, protocol id, execution profile, network
   profile, load profile, load shape, repetition). They do not depend on
   execution timing, host state, or tool behavior.
2. **Preservation.** Raw tool output is always preserved. Parsed metrics
   are supplementary, not a replacement.
3. **Completeness.** Every artifact that can be captured has a defined path.
   Missing artifacts (e.g., qlog when the target does not export it) are
   expected and do not indicate failure.
4. **Separation.** Validation artifacts, protocol proof artifacts, and
   benchmark artifacts live in separate files. A single result is not asked
   to represent both functional correctness and performance.
5. **Auditability.** Command lines, Docker commands, inspect output, and
   metadata files allow a consumer to reproduce or audit the execution
   environment.

## Proposed Extensions

- **Artifact retention policies:** Configuration for which artifacts to keep,
  compress, or discard after a run.
- **Artifact signing:** Cryptographic signatures on result files to prove
  they were produced by a specific runner instance.
- **Artifact upload:** Push artifacts to a remote store (S3, blob storage)
  as part of a CI or hosted execution pipeline.
- **Run index federation:** Link run indices across runs to enable trend
  analysis and regression detection.
